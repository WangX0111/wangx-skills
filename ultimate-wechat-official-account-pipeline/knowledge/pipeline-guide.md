## 三种工作模式

### Mode A: 从零创作
用户只有选题/想法/零散素材，需要完整的内容创作流程。

### Mode B: 内容优化
用户已有内容（草稿/文章/笔记/PDF/链接），需要基于原内容进行改写、优化、降味、排版、发布。

### Mode C: 内容重构
用户输入链接/多个链接或内容，要求深度改写、重构、完全重建表达，产出全新原创文章。

> 改写原则、叙事模式、重构红线等详细规范见 [references/mode-c-reconstruction.md](../references/mode-c-reconstruction.md)

## Mode A: 从零创作 - 五阶段流水线

### Phase 1: 智能选题策划

**输入类型识别**：
| 输入 | 处理方式 |
|:-----|:---------|
| 主题词 | 热点追踪 + 竞品分析 |
| 链接 | 内容解析 + 角度重构 |
| 语音/想法 | 信息结构化 |
| PDF/Brief | 核心信息提取 |

**选题评分矩阵（HKR+Data）**：
- **H (Happy)**：有趣？有悬念？标题吸引力？
- **K (Knowledge)**：有信息量？有认知增量？
- **R (Resonance)**：能戳中情绪？能引发共鸣？
- **Data**：搜索热度、竞品覆盖度、时效性、账号调性匹配

**通过标准**：至少满足HKR中的两项。

### Phase 2: AI协作内容创作

**风格引擎加载**：
根据账号配置加载对应风格预设。

**文章原型选择**：
| 原型 | 判断标准 |
|:-----|:---------|
| 调查实验型 | 用户说"我去试了/测了/体验了" |
| 产品体验型 | 围绕某个产品/工具的使用 |
| 现象解读型 | 观察到一个现象要分析背后 |
| 工具分享型 | 想推荐一个工具/Prompt |
| 方法论型 | 想分享经验/心得/方法 |

**创作流程**：
```
加载风格 → 确定原型 → 生成大纲 → 用户确认 → 分段扩写 → 四层质检 → 输出
```

**人机协作边界**：
- AI做：找论据、找类比、扩写、补充背景、梳理结构
- 人做：真实经历、核心角度、情绪表达、同理心转换

### Phase 3: 智能视觉设计

**配图策略**：
- 自动标记配图位置（每300字1张）
- 生成结构化图片描述（illustration_spec）
- 调用AI绘图或推荐无版权图片

**封面设计**：
- 根据文章类型和情感色彩推荐风格
- 生成多种封面供选择

**排版主题**：
- 根据文章类型智能推荐主题
- 应用主题生成微信兼容HTML

### Phase 4: 发布运营

**发布前终审清单**：
- [ ] 内容质量 ≥ 70分
- [ ] 四层质检通过
- [ ] 配图完整
- [ ] 封面图正确（900×383px）
- [ ] 元数据完整（标题/摘要/作者）
- [ ] 敏感词扫描通过
- [ ] 用户确认

**发布通道选择**：
- API通道：快速、支持定时
- Browser通道：无需API配置

### Phase 5: 数据追踪

- 阅读量、点赞、在看、分享
- 与历史文章对比
- 效果简报
- 迭代建议

## Mode B: 内容优化 - 五步骤流程

### Step 1: 内容诊断

分析维度：
- 结构完整性（开头/逻辑/结尾）
- 信息密度（空泛度/支撑度）
- 风格评估（AI味1-5级/口语化/活人感）
- 平台适配性
- 优化潜力

输出：诊断报告（🔴严重/🟡建议/🟢优秀）

### Step 2: 优化方案确认

| 方案 | 适用场景 |
|:-----|:---------|
| 轻量润色 | 整体不错，局部优化 |
| 深度改写 | 结构或风格需大调整 |
| 扩写丰富 | 骨架好但太单薄 |
| 精简提炼 | 冗余、重点不突出 |
| 风格迁移 | 需要切换写作风格 |
| 降味处理 | AI痕迹明显 |

用户选择方案后进入下一步。

### Step 3: 重新创作

- 保留原内容核心价值
- 应用选定风格引擎
- 按方案执行补充/删减
- 人机协作：用户注入真实经历

### Step 4: 对比输出

展示原文 vs 优化版 + 改动说明。

### Step 5: 用户终审

用户确认后进入 Phase 3 视觉设计。

## Mode C: 内容重构 - 六步骤流程

> 详细改写原则和规范见 [references/mode-c-reconstruction.md](../references/mode-c-reconstruction.md)

### Step 1: 深度分析

对每个链接或内容进行深度分析，提取：

| 分析维度 | 提取内容 |
|:---------|:---------|
| 核心论点 | 文章最重要的3-5个结论 |
| 关键论据 | 支撑核心论点的数据、实验、推理 |
| 论证链条 | 论点之间的逻辑关系和推导路径 |
| 技术细节 | 关键技术指标、方法名称、具体参数 |
| 内容脉络 | 整体叙事结构和信息流转方式 |
| 可补充空间 | 原文未展开但重要的背景和上下文 |

**多链接特殊处理**：
- 交叉验证：不同来源对同一论点的表述差异
- 信息整合：互补信息合并，重叠信息去重
- 矛盾标注：不同来源的矛盾之处，标注供用户确认

输出分析报告，向用户确认核心信息提取是否完整准确。

### Step 2: 结构设计

1. 选择叙事模式：
   - 问题驱动型：问题定义 → 方案推导 → 设计原则 → 模块解释 → 意义总结
   - 挑战突破型：现状分析 → 挑战识别 → 解决思路 → 实现路径 → 效果评估
   - 脉络梳理型：背景铺垫 → 核心创新 → 技术细节 → 应用场景 → 未来展望
   - 逆向解构型：效果/结论先行 → 逐层回溯原因 → 揭示设计智慧
2. 设计新的标题层级和段落组织
3. 确定开头切入点和结尾收束方式
4. 规划核心论点的出现时机和论证节奏
5. 标注需要补充的背景信息位置

向用户展示结构方案，确认后进入 Step 3。

### Step 3: 全文重构

- 加载风格引擎
- 按新结构逐段重写（彻底重建表达，禁止近义词替换）
- 补充背景解释和上下文
- 运行四层质检（L1-L4）+ Mode C 专项质检（详见 quality-checklist.md）
- 独立性检查：验证文章可脱离原文独立阅读

**重构红线**（详见 references/mode-c-reconstruction.md）：
- 禁止近义词替换式改写
- 禁止保留原文标志性表达
- 必须保留核心论点和关键论据
- 必须保留完整论证链条

### Step 4: 对比展示

展示原文 vs 重构版 + 重构说明：
- 结构变化
- 叙事变化（切入点转换）
- 表达重建（句式/比喻/修辞完全重建）
- 信息保留（核心论点完整保留）
- 信息增强（补充的背景解释）

### Step 5: 深度分析（作者判断章节）

在文章结尾加入独立分析章节：

| 分析维度 | 内容 |
|:---------|:-----|
| **真正贡献** | 这项工作的真正核心贡献是什么，解决了什么关键问题 |
| **证据不足** | 目前证据还不充分或存在疑虑的地方，有哪些局限性 |
| **后续验证** | 后续最值得进行的消融实验或验证工作，下一步研究方向 |

> 此章节为原创判断，非对原文观点的复述。

### Step 6: 用户终审

用户确认满意后进入 Phase 3 视觉设计；有修改意见则迭代优化。

## 质量门禁体系

每个Phase的通过标准：

| Phase | 门禁标准 | 未通过处理 |
|:------|:---------|:-----------|
| Phase 1 | HKR至少2项通过 | 建议调整选题 |
| Phase 2 | 四层质检通过 + 评分≥70 | 返回修改 |
| Phase 3 | 配图完整性 + 封面存在 | 补充视觉素材 |
| Phase 4 | 终审清单全通过 + 用户确认 | 继续完善 |
| Phase 5 | 发布成功 | 排查问题 |

## 输出目录处理

### 确定输出目录

Pipeline 启动时，第一步确定输出目录：

1. 检查用户是否在指令中指定了输出位置（如「保存到 ~/Desktop/」）
2. 检查输入文件的 frontmatter 中是否有 `output_dir` 字段
3. 检查当前账号配置的 `output_dir`
4. 检查全局默认的 `output_dir`
5. 自动推断：基于文章标题生成 slug，输出到 `./wechat-pipeline/{slug}/`

确定输出目录后，**立即告知用户**：

```
📁 输出目录：/path/to/wechat-pipeline/ai-coding-assistant/
```

### 各阶段产出文件

| Phase | 产出文件 | 保存位置 |
|:------|:---------|:---------|
| Phase 2 | 文章 Markdown | `{output_dir}/article.md` |
| Phase 3 | 封面图 | `{output_dir}/cover.png` |
| Phase 3 | 正文配图 | `{output_dir}/images/01-{slug}.png` |
| Phase 3 | 排版 HTML | `{output_dir}/article.html` |
| Phase 4 | 发布元数据 | `{output_dir}/meta.yaml` |

### 目录冲突处理

如果输出目录已存在上次产出：
1. 告知用户目录已存在
2. 提供选项：覆盖 / 使用新目录（追加时间戳） / 取消
3. 用户选择后继续

## 常见问题

**Q: Mode A、Mode B、Mode C 怎么区分？**
A: 根据用户输入和意图自动判断：
- 只有选题/想法 → Mode A（从零创作）
- 有已有内容，要求优化/降味/润色 → Mode B（内容优化）
- 有链接/内容，要求深度改写/重构/完全重建 → Mode C（内容重构）

**Q: Mode B和Mode C的核心区别是什么？**
A: Mode B保留原文结构骨架做局部优化，Mode C彻底重建表达方式和文章结构。如果用户说「优化一下」「降味」→ Mode B；如果说「深度改写」「重构」「用自己的话重写」→ Mode C。

**Q: 优化时会不会丢失原内容的核心信息？**
A: 不会。Mode B和Mode C的第一原则都是「保留核心价值」，所有改动以不丢失原内容关键信息为前提。

**Q: 可以多账号同时管理吗？**
A: 可以。在accounts.yaml中配置多个账号，发布时自动提示选择。
